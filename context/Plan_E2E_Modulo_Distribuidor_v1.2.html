1. Propósito y alcance

Este documento guía al equipo de Motorambar en la ejecución de las pruebas end-to-end del Módulo Distribuidor del Portal de Vehículos. Organiza los escenarios en cuatro fases de prueba y, dentro de cada fase, por módulo funcional. Cada escenario se presenta en formato checklist: el ejecutor realiza el escenario, compara contra el resultado esperado y marca el estatus.

Alcance funcional (según las pantallas del portal):

  • Dashboard Ejecutivo: KPIs, gráficas, Exportar Datos y Descargar Excel
  • Vehículos: Vehículos Importados, Importar Vehículos, Historial de Importaciones
  • CPA: Importar CPA (batch PDF con cruce por VIN, Unlinked y Re-link) e Historial de CPA
  • Grid de Vehículos Importados: búsqueda y filtros, vistas, detalle expandible (More Info, CO, CPA, Factura, Additional Docs, History)
  • Acciones individuales: Preview, descarga, Add Additional Document, Edit Vehicle, Assign To, Unassign, Manual Release, Delete
  • Acciones por lote (barra de selección): Download (N), Assign To, Unassign, Upload Doc., Manual Release, Resend Data to PDV, Send Docs to PDV, Delete
  • Integración PDV: envío de datos y documentos, reintentos, indicadores PDV-Data / PDV-Documents
  • CO: generación, edición de campos y regeneración con nuevos valores
  • Reportes: Generate Report, Exportar Datos, Download History
  • Transversales: roles, idioma (ES/EN), Dark Theme, notificaciones, sesión, auditoría
Nota: Los escenarios marcados "si aplica" dependen de reglas de negocio a confirmar. El KPI del Dashboard identificado como "[Pending to define]" debe definirse antes del sign-off.

2. Instrucciones de ejecución

  • Ejecutar las fases en orden: la Fase 1 genera los datos que usan las Fases 2 y 3; la Fase 4 se ejecuta al final.
  • Registrar el estatus de cada escenario: Pass, Fail (abrir defecto), Blocked (dependencia o defecto previo) o N/A (justificado).
  • Para cada Fail, anotar en Comentarios el número del defecto y adjuntar evidencia (captura, archivo usado, mensaje de error).
  • Guardar los archivos de prueba utilizados (archivos de importación, PDF de CPA, documentos cargados) como evidencia.
  • Ejecutar cada escenario con el rol indicado; los escenarios de permisos requieren usuarios de prueba con distintos perfiles.
  • Ejecutar los escenarios de idioma y Dark Theme al menos una vez sobre cada pantalla principal.

3. Prerrequisitos y datos de prueba

  • Ambiente de pruebas (QA/UAT) con la versión candidata del Portal Distribuidor instalada y estable.
  • Usuarios de prueba por rol (Distribuidor y demás perfiles definidos), incluyendo el usuario Motorambar.
  • Set de archivos de importación de vehículos: válido, con errores controlados, con duplicados, corrupto, layout inválido y que exceda el tamaño máximo.
  • PDF de CPA multipágina real o representativo (ej. CPA Nissan) incluyendo páginas que no crucen por VIN para probar Unlinked/Re-link.
  • Ambiente PDV (o mock) disponible para probar envío de datos y documentos, incluyendo forzar un fallo de envío.
  • Catálogos y parámetros configurados (localidades, instituciones para Assign To, estatus, valores permitidos de campos CO).
  • Canal acordado para el registro de defectos y el repositorio de evidencias.

4. Fase 1 — Importación de datos e historiales

Valida los mecanismos de entrada de datos: importación automatizada, Importar Vehículos por pantalla, Importar CPA (batch PDF con cruce por VIN) y los historiales de importaciones y de CPA. Es prerrequisito de las fases siguientes.

1.1 Importación de Datos Automatizada

ID Escenario Resultado esperado Estatus Comentarios
IMPAUT-01 El proceso automatizado se ejecuta en el horario/frecuencia configurado y toma el archivo de la ubicación acordada (SFTP/carpeta). El job corre según calendario, procesa el archivo y la corrida queda registrada en el Historial de Importaciones.
IMPAUT-02 Importación de archivo válido con múltiples registros de vehículos. Todos los registros se cargan y los vehículos aparecen en Vehículos Importados con sus datos correctos; los contadores del Dashboard se actualizan.
IMPAUT-03 Archivo con registros válidos e inválidos (campos requeridos vacíos, formatos incorrectos). Los válidos se importan; los inválidos se rechazan con detalle del error por registro en el historial/log.
IMPAUT-04 Archivo con VIN duplicados (ya existentes en el portal). El sistema detecta el duplicado, no crea doble registro y lo reporta.
IMPAUT-05 Archivo corrupto, vacío o con estructura/layout incorrecto. El proceso rechaza el archivo completo, registra el error y notifica; no hay carga parcial inconsistente.
IMPAUT-06 Falla de conexión o archivo no disponible a la hora del proceso. El sistema registra el fallo, notifica y permite el reintento sin duplicar datos.
IMPAUT-07 Reejecución del proceso con el mismo archivo ya procesado. No se duplican registros (idempotencia) y queda evidencia en el historial.

1.2 Importar Vehículos (por pantalla)

ID Escenario Resultado esperado Estatus Comentarios
IMPPAN-01 Acceder por el menú Vehículos → Importar Vehículos y cargar un archivo válido. El archivo se valida, se confirma con totales (leídos, cargados, rechazados) y los vehículos aparecen en Vehículos Importados.
IMPPAN-02 Verificar los datos cargados en el detalle del vehículo (tab More Info). Vehicle Information (VIN, Year, Maker, Model), Technical Data (Auto Color, Doors, Pistons, Horsepower) e IDs and Weight (Propulsion Type, Unloaded Weight, Load Capacity, Vehicle Unit) coinciden con el archivo.
IMPPAN-03 Carga de archivo con formato no permitido. El sistema bloquea la carga e indica los formatos aceptados.
IMPPAN-04 Carga de archivo que excede el tamaño máximo. El sistema rechaza el archivo indicando el límite.
IMPPAN-05 Archivo con columnas faltantes o encabezados incorrectos. El sistema identifica el problema de layout, no procesa y muestra el detalle.
IMPPAN-06 Archivo con VIN duplicados respecto a datos existentes. El sistema informa los duplicados y no los vuelve a crear.
IMPPAN-07 Cancelación de la importación antes de confirmar. No se carga ningún dato.
IMPPAN-08 Usuario sin permisos intenta importar. El acceso es denegado según el rol.

8. Criterios de entrada y salida

Criterios de entrada

  • Prerrequisitos de la Sección 3 cumplidos.
  • Smoke test básico exitoso (login, navegación a Dashboard, Vehículos y CPA).
  • Datos y archivos de prueba cargados y validados.

Criterios de salida (sign-off)

  • 100% de los escenarios ejecutados (Pass, Fail documentado, o N/A justificado).
  • 0 defectos críticos o altos abiertos; defectos medios/bajos con plan de acción acordado.
  • Escenarios end-to-end de la Fase 4 en Pass.
  • KPI "[Pending to define]" del Dashboard definido o formalmente excluido.
  • Evidencias completas y resumen de ejecución firmado por los aprobadores.

9. Gestión de defectos y evidencias

Todo Fail debe registrarse como defecto con: ID del escenario, descripción, pasos para reproducir, resultado actual vs. esperado, severidad (crítica / alta / media / baja), evidencia adjunta y ambiente. Los defectos críticos detienen la fase afectada hasta su corrección; tras cada corrección se re-ejecuta el escenario y los escenarios relacionados (regresión dirigida).

10. Proceso de estabilización: Hot Fixes y Change Requests

10.1 Hot Fixes

Se considerarán como hot fixes aquellos ajustes o correcciones que estén relacionados directamente con el alcance inicial acordado del proyecto. Estos serán atendidos como parte del proceso de estabilización, siempre que correspondan a funcionalidades previamente definidas.

10.2 Mejoras o Change Requests

Todo requerimiento, ajuste o funcionalidad que no forme parte del alcance inicial será tratado como una mejora o change request. Estas mejoras deberán ser discutidas internamente por Motorambar para validar prioridad, necesidad operacional e impacto. Luego, deberán ser discutidas con Autoreg para su evaluación, estimación y planificación correspondiente.

10.3 Evidencia requerida

Para facilitar el análisis y seguimiento, todo hallazgo deberá venir acompañado de evidencia, comentarios claros y, cuando aplique, fotos o videos que permitan validar el comportamiento reportado. Esta información será utilizada como anejo y evidencia para documentar cada caso correctamente.

10.4 Acompañamiento semanal

Como parte del apoyo al proceso, se realizarán visitas semanales a Motorambar con el objetivo de continuar fortaleciendo el uso del sistema, atender dudas operacionales y lograr una madurez completa de la plataforma.

13. Recomendaciones adicionales

  • Congelar la versión durante el ciclo de pruebas: cualquier despliegue nuevo al ambiente debe comunicarse y puede requerir re-ejecutar escenarios ya pasados.
  • Probar el batch CPA con los PDFs reales de Motorambar (volumen y calidad de escaneo reales), ya que el cruce por VIN depende de la lectura del documento.
  • Medir tiempos en los escenarios de volumen (importaciones grandes, batch CPA de 50+ páginas, descargas y asignaciones por lote, reportes) y acordar umbrales con el negocio.
  • Probar la integración PDV en ambos sentidos: envío exitoso, fallo y reintento, verificando que no se dupliquen casos en PDV.
  • Incluir una sesión de UAT exploratoria con usuarios finales de Motorambar al terminar el checklist.
  • Verificar el comportamiento de las notificaciones del portal (campana) asociadas a importaciones, CPA y envíos a PDV.
  • Mantener este documento como base de regresión para futuras versiones, actualizando los escenarios cuando cambien las reglas de negocio.
